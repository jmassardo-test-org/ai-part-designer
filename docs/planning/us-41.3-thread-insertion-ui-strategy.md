## 1. User Story & Acceptance Criteria

### Primary User Story

**As a** mechanical designer working in the AssemblematicAI 3D viewport,
**I want** a thread insertion UI that lets me select a thread family, configure sizing and direction, optionally apply print optimization, and preview the result in 3D before applying,
**So that** I can add accurately specified threads to my part designs without manually entering geometry parameters.

---

### AC-1: ThreadFamilySelector — Card Grid Rendering

```
GIVEN the Thread Wizard is open and on Step 1
WHEN the thread families API returns data
THEN a 2-column card grid displays one card per family (ISO Metric, UNC, UNF, NPT, BSPP, BSPT, ACME, Trapezoidal)
AND each card shows the family name, a description (truncated to 2 lines), and a badge with the size count
AND the ISO Metric card displays a "★ Common" indicator
AND clicking a card highlights it with a primary border + ring
AND the "Next" button becomes enabled only after a family is selected
```

### AC-2: ThreadFamilySelector — Loading & Error States

```
GIVEN the wizard opens on Step 1
WHEN the families API is loading
THEN 6 skeleton cards are rendered in a 2-column grid
AND each skeleton card is 80px tall with rounded corners

GIVEN the families API returns an error
WHEN the user views Step 1
THEN a destructive-styled alert is shown with the message "Failed to load thread families. Please try again."
AND a "Retry" button is present that re-triggers the families query
```

### AC-3: ThreadFamilySelector — Keyboard Navigation

```
GIVEN the family card grid is rendered
WHEN the user presses Tab
THEN focus moves to the next family card in DOM order
AND the focused card has a visible focus ring (outline-2 outline-offset-2)

WHEN the user presses Enter or Space on a focused card
THEN that family is selected (same as click)

WHEN the user presses arrow keys (Up/Down/Left/Right)
THEN focus moves between cards in the grid following a 2-column layout pattern
```

### AC-4: ThreadSizeSelector — Pitch Series Toggle (Metric Only)

```
GIVEN the user selected "ISO Metric" on Step 1 and is on Step 2
WHEN the size selector renders

AND the size selection resets to null when pitch series changes

GIVEN the user selected a non-metric family (UNC, UNF, NPT, etc.)

THEN the pitch series toggle is hidden
```


### AC-10: PrintOptimizationForm — Optional Step

```
GIVEN the user is on Step 3 (Configure)
WHEN the form renders
THEN an "Enable Print Optimization" toggle/checkbox is shown below the base configuration
AND when disabled (default), the print optimization fields are hidden
AND when enabled, the following fields appear:

  - Adjusted Major Diameter (computed from clearance)

AND a feasibility badge is shown: "Excellent" (green), "Good" (blue), "Marginal" (yellow), "Not Recommended" (red)
```

### AC-11: PrintOptimizationForm — Feasibility Warnings

```
GIVEN print optimization is enabled
WHEN the selected thread size is below recommended minimums (M2 for FDM, M3 for SLA)
THEN a warning alert appears: "This thread size is below the recommended minimum for {process}. Consider using a heat-set insert."
AND the feasibility badge shows "Not Recommended" or "Marginal"
AND the user can still proceed (warning, not blocking)

GIVEN the selected family is NPT or BSPT (tapered)
WHEN print optimization is enabled
THEN a warning alert appears: "Tapered pipe threads are not suitable for functional 3D prints. For printed fluid connections, consider O-ring grooves."
```

```
AND a fallback message appears if WebGL is not available: "3D preview unavailable. Your browser does not support WebGL."
```

### AC-17: Wizard State Management — Navigation Integrity

```
GIVEN the user is mid-way through the wizard (e.g., on Step 3)
WHEN the user navigates back to Step 1 and selects a different family
THEN the selected size resets to null (Step 2 must be re-done)
AND the print optimization settings are preserved
AND the generation result (if any) is cleared

GIVEN the user closes the wizard (Cancel or X) at any step
WHEN the wizard re-opens
THEN all state is reset to defaults (family=null, size=null, type=external, hand=right, length=20, chamfer=true, print optimization disabled)
```

```
GIVEN the user clicks "Next" or "Back"
```

```
GIVEN the user has progressed past Step 1 (i.e., has selected a family at minimum)

AND "Continue Editing" returns to the wizard
AND "Discard" closes the wizard and resets state

GIVEN the user is on Step 1 with no selection
WHEN closing the wizard
THEN no confirmation prompt is shown — the wizard closes immediately
```

## 3. UI/UX Design Specifications

**Layout:**
- Container: `grid grid-cols-2 gap-3` (desktop), `grid grid-cols-1 gap-3` (mobile, `sm:` breakpoint)
- Card: `rounded-lg border p-4 text-left transition-colors hover:bg-accent cursor-pointer`
- Selected card: `border-primary bg-primary/5 ring-1 ring-primary`
- Focus: `focus-visible:outline-2 focus-visible:outline-primary focus-visible:outline-offset-2`

- Line 1: Family name — `font-medium text-sm`

- Each card: `role="radio"`, `aria-checked={selected}`, `aria-label="{name} - {description}"`

### 3.5 Wizard Step Transitions

**Implementation:** Use CSS transitions on the step content wrapper:

```tsx
<div className={`transition-opacity duration-150 ease-in-out ${
  transitioning ? 'opacity-0' : 'opacity-100'
}`}>
  {/* step content */}
</div>
```

On step change:
1. Set `transitioning = true` → content fades out (150ms)
2. After 150ms timeout, swap the step content
3. Set `transitioning = false` → content fades in (150ms)

**Progress bar:** Already implemented with colored bars. The transition between bar states uses `transition-colors` (already present).

## 4. Edge Cases & Error Handling

### 4.1 API Failures Mid-Wizard

| Scenario | Behavior |
|----------|----------|
| **Families API fails (Step 1)** | Show destructive alert with "Retry" button. User cannot proceed. `useThreadFamilies` uses default React Query retry (3 attempts with exponential backoff). |
| **Sizes API fails (Step 2)** | Show destructive alert in the size area. Family selection is preserved. "Retry" button re-fetches sizes. User can go back to Step 1, which remains functional. |
| **Spec API fails (Step 2)** | The spec panel shows "Failed to load specification." inline. Size selection still works, and the user can proceed to Step 3 without the spec preview. Generation will still work because it sends family+size directly. |
| **Tap drill API fails (Step 2)** | `TapDrillReference` already handles this — shows a destructive alert (existing implementation). Non-blocking. |
| **Print recommendation API fails (Step 3)** | The feasibility badge and orientation recommendation are hidden. A subtle inline message says "Print recommendations unavailable." The user can still configure print optimization manually. |
| **Generate mutation fails (Step 4)** | Show destructive alert with error message. Show "Retry" button. "Back" button stays enabled so user can adjust parameters. Error message truncated to 200 chars to avoid layout breakage. |
| **Generate mutation times out (>60s)** | The API returns a 504/408. The error is displayed as "Thread generation timed out. Try a shorter thread length or simpler configuration." Retry button available. |

### 4.2 Invalid Combinations

| Scenario | Behavior |
|----------|----------|
| **Family selected, then API data changes** | If the sizes query returns and the previously selected size no longer exists, reset size to null and show info toast: "Size selection cleared — available sizes have changed." |
| **Length = 0 or negative** | The number input has `min={1}`. If the user types 0 or negative, the "Next" button is disabled (`canProceed` returns false when `lengthMm <= 0`). Input validation message: "Length must be between 1 and 500 mm." |
| **Length > 200mm with print optimization** | Show a warning: "Threads longer than 200mm may require extended generation time (up to 60 seconds)." Non-blocking. |
| **Custom clearance ≤ 0 or > pitch/2** | Validated on the input field. Error message: "Clearance must be between 0.05mm and {pitch/2}mm." |
| **Left-hand + NPT** | NPT threads are almost never left-hand. Show info banner: "Left-hand NPT threads are non-standard. Ensure your mating part matches." Non-blocking. |

### 4.3 3D Preview Failures

| Scenario | Behavior |
|----------|----------|
| **WebGL not supported** | Error boundary around Canvas catches the error. Renders a static 2D SVG cross-section with key dimensions annotated. Message: "3D preview requires WebGL. Showing 2D reference instead." |
| **STL data fails to parse** | Catch the error in the STL loader. Show the generation result metadata (face count, generation time) without the 3D preview. Message: "3D preview failed to load. Thread was generated successfully." |
| **Canvas crashes (GPU driver issue)** | Error boundary catches. Same fallback as WebGL failure. Log the error to the console and to monitoring. |
| **Very large thread (>100k faces)** | Before rendering, check `estimated_face_count` from the response. If > 100k, show a warning: "This is a complex thread. 3D preview may be slow." Offer a "Load Preview" button instead of auto-rendering. If > 500k, skip 3D preview entirely and show metadata only. |

### 4.4 Thread Generation Timeout

| Scenario | Behavior |
|----------|----------|
| **< 10s** | Normal flow — spinner with "Generating…" |
| **10–30s** | Spinner updates to "Still generating… Large threads may take up to 60 seconds." |
| **30–60s** | Spinner updates to "Almost there… This is a complex thread." |
| **> 60s (API timeout)** | API returns timeout error. Show: "Generation exceeded the time limit. Suggestions: reduce thread length, use a larger pitch, or try a coarser tolerance." |

### 4.5 User Navigates Away Mid-Wizard

| Scenario | Behavior |
|----------|----------|
| **Close wizard (X/Cancel/Backdrop)** | Confirmation dialog if any state beyond Step 1 selection exists (see AC-19). Wizard state is reset on confirmed close. |
| **Browser navigation (e.g., back button)** | The wizard is a modal, not a route. Browser navigation closes the dialog naturally. The `onClose` handler is called, which resets state. No risk of orphaned state. |
| **Generation in progress + close** | The `generateMutation` is not cancelled (React Query mutation). The backend will complete. The result is discarded client-side. No cleanup needed. Future enhancement: `AbortController` support. |
| **Session expires mid-wizard** | API calls will return 401. The global auth error handler redirects the user to login. Wizard state is lost. No special handling needed beyond what the app already does. |

### 4.6 Concurrent Wizard Instances

Only one wizard instance can be open at a time. The wizard is controlled by a single `isOpen` boolean. If the user opens the wizard from the toolbar and then navigates to the component library (without closing it), the modal persists. This is standard React modal behavior. No special handling needed.

## 6. Security Considerations

Per the project's mandatory security review checklist:

### Authentication & Authorization
- The wizard calls authenticated API endpoints (`useGenerateThread` sends Bearer token via `useAuth()`)
- All `POST /api/v2/threads/generate` endpoints require `current_user: User = Depends(get_current_user)` (already implemented in backend)
- `GET` endpoints (families, sizes, spec) require standard API authentication (already implemented)
- No new endpoints are introduced by this UI story — all backend endpoints are in place

### Input Validation & Sanitization
- **Family/size values** are selected from API-provided lists (not free-text), preventing injection
- **Length** is validated client-side (`min=1, max=500`) and server-side (`Field(gt=0, le=500)`)
- **Custom clearance** is validated client-side and server-side (`Field(ge=0.05, le=2.0)`)
- **ThreadType and ThreadHand** are constrained to enum values via TypeScript types and server-side regex patterns
- No user-provided strings are rendered as raw HTML (React's JSX auto-escapes)

### Rate Limiting & Throttling
- Thread generation triggers `POST` to a rate-limited endpoint (10 req/min per user — already configured)
- The "Generate" button is disabled while `isPending` is true, preventing rapid re-submission
- The "Retry" button does not bypass rate limiting

### Logging & Monitoring
- All generation requests are logged server-side with user ID, parameters, timing (already implemented)
- Client-side: log generation errors to console and monitoring (e.g., Sentry) if configured
- No new security-sensitive logging is introduced on the frontend

### Content Security
- The 3D preview uses React Three Fiber with geometry from the API. STL data is binary — no script injection vector.
- Download URLs are constructed server-side and returned via the API response. The frontend uses them as `href` — no path traversal risk.
- The ThreadWizard does not persist any data to localStorage/sessionStorage (state is in-memory React state only)
