## 1. Executive Summary

### Current State

The Thread Wizard (`ThreadWizard.tsx`, 607 lines) is a fully functional 4-step dialog (family → size → configure → generate). Supporting layers are complete: TypeScript types (`threads.ts`), API client (`lib/api/threads.ts`), React Query hooks (`useThreads.ts`), and `TapDrillReference.tsx`.

### Gaps to Close

| Gap | Description | Severity |
|-----|-------------|----------|
| **G1** | No `PrintOptimizationForm` — the backend's `/threads/generate/print-optimized` endpoint and `usePrintRecommendation` / `useGeneratePrintOptimized` hooks exist but have no corresponding UI | Feature gap |
| **G2** | No `ThreadPreview3D` — no 3D thread preview; generation step only shows a text summary | Feature gap |
| **G3** | No toolbar entry point — AdvancedCADViewer has no "Add Thread" button; the wizard can only be opened programmatically | Integration gap |
| **G4** | No Component Library entry — `ComponentLibraryPage` has no "Threads" category | Integration gap |
| **G5** | Missing pitch series selector — `StepSize` has no coarse/fine toggle for ISO Metric threads despite the `useThreadSizes` hook accepting `pitchSeries` | Partial implementation |
| **G6** | Test coverage incomplete — no tests for print optimization, 3D preview, toolbar integration, or pitch series toggle | Quality gap |

### Approach

- **Additive, not rewrite.** Introduce three new files (PrintOptimizationForm, ThreadPreview3D, index barrel) and modify three existing files (ThreadWizard, AdvancedCADViewer, ComponentLibraryPage). No refactoring of StepFamily/StepSize into separate files — the inline sub-components in ThreadWizard already work and match the existing pattern.
- **Print optimization is embedded in Step 3 (Configure)**, not a new step. This avoids breaking the 4-step progress bar, the existing test suite, and the AC-10 acceptance criterion that specifies it as a section within the Configure step.
- **ThreadPreview3D is embedded in Step 4 (Generate)**, shown after successful generation.

### 2.2 ThreadPreview3D (NEW)

**File:** `frontend/src/components/threads/ThreadPreview3D.tsx`

**Purpose:** React Three Fiber canvas displaying a simplified thread visualization with dimension annotations. Shown in the Generate step after a successful generation.

#### Props Interface

```typescript
interface ThreadPreview3DProps {
  /** Thread specification with diameters and pitch. */
  spec: ThreadSpec;
  /** Thread type (internal/external). */
  threadType: ThreadType;
  /** Thread length in mm. */
  lengthMm: number;
  /** Whether the thread has been generated successfully. */
  isGenerated: boolean;
  /** Optional CSS class for the container. */
  className?: string;
}
```

#### Architecture Notes

- **Geometry approach:** Render a parametric cylinder with helical groove geometry generated client-side. The component does NOT load an STL from the server — it creates a simplified preview from the `ThreadSpec` dimensions:
  - `major_diameter` → outer cylinder radius
  - `minor_diameter_ext` (for external) or `minor_diameter_int` (for internal) → groove depth
  - `pitch_mm` → helix spacing
  - `lengthMm` → cylinder height
- **Material:** `<meshStandardMaterial color="#94a3b8" metalness={0.7} roughness={0.3} />` — silver metallic, matching the design spec.
- **Annotations:** Use `@react-three/drei` `Html` component to overlay 3 dimension labels:
  - Horizontal arrow: major diameter (e.g., "Ø8.000")
  - Vertical tick marks: pitch (e.g., "P 1.25")
  - Vertical bracket: total length (e.g., "L 20.0")
- **Orbit controls:** `<OrbitControls enableDamping dampingFactor={0.1} />` — users can rotate/zoom the preview.
- **Lighting:** Ambient (0.5) + directional (position `[10, 10, 5]`, intensity 1.0).
- **Grid:** `<Grid args={[100, 100]} />` from drei.
- **Camera:** `position={[0, 0, majorDiameter * 4]}`, `fov={45}` — auto-scales based on thread size.

#### WebGL Fallback

Wrap the `<Canvas>` in a React error boundary. On WebGL initialization failure:

```
<div data-testid="thread-preview-fallback" class="flex items-center justify-center ...">
  <p>3D preview requires WebGL. Showing specification table instead.</p>
  <!-- Render spec table as static fallback -->
</div>
```

The error boundary component is defined inline in this file (not shared) because it is tightly coupled to the R3F context.

#### Existing R3F Patterns Referenced

- **`AdvancedCADViewer.tsx`** — Canvas setup, OrbitControls ref, Suspense fallback with `Html`, lighting rig, grid configuration.
- **`LayoutPreview3D.tsx`** — Simpler canvas with OrbitControls, center, grid. Test mocking pattern for `@react-three/fiber` and `@react-three/drei`.

## 5. Security Controls

### 5.1 Input Validation — PrintOptimizationForm

All numeric inputs must be validated both at the HTML constraint level (min/max attributes) and programmatically before submission.

| Field | Type | Min | Max | Default | Validation Rule |
|-------|------|-----|-----|---------|-----------------|
| `nozzle_diameter_mm` | number | 0.1 | 2.0 | 0.4 | Must be positive float; step 0.05; reject NaN |
| `layer_height_mm` | number | 0.01 | 1.0 | 0.2 | Must be positive float; step 0.01; reject NaN; must be ≤ nozzle_diameter_mm |
| `custom_clearance_mm` | number \| null | 0.01 | 5.0 | null | Only validated when override is enabled; must be positive float; reject NaN |
| `length_mm` (existing) | number | 1 | 500 | 20 | Already validated in StepConfigure; no change needed |

**Implementation:** Each input uses `type="number"` with `min`, `max`, and `step` HTML attributes. The `onChange` handler parses with `Number()` and clamps to bounds. The `canProceed()` function in the wizard validates all values before enabling "Next"/"Generate".

**Defense against NaN/Infinity:**
```typescript
const sanitizeNumeric = (value: string, min: number, max: number, fallback: number): number => {
  const n = Number(value);
  if (!Number.isFinite(n)) return fallback;
  return Math.max(min, Math.min(max, n));
};
```

This utility should be defined in the `PrintOptimizationForm.tsx` file (not shared) since it's specific to this form's validation needs.

### 5.2 XSS Prevention

- **No `dangerouslySetInnerHTML`:** All thread data (family names, descriptions, spec values, recommendation notes, orientation advice) is rendered via JSX text content, which React auto-escapes.
- **No URL construction from user input:** Thread family IDs and size strings are used in API paths via `encodeURIComponent()` (already implemented in `lib/api/threads.ts`).
- **No `eval` or dynamic code execution:** Thread preview geometry is built from numeric spec values only.
- **API response data:** The `notes[]` array from `PrintRecommendation` is rendered as a `<ul>` list with each note as escaped text content. The `orientation_advice` string is rendered as paragraph text.

### 5.3 Authentication & Authorization

- **Auth token handling:** All API calls go through the hooks in `useThreads.ts`, which extract the token from `useAuth()` context and pass it to the API client as a Bearer token in the `Authorization` header. No changes needed — this is already implemented.
- **Protected routes:** The AdvancedCADViewer and ComponentLibraryPage are already behind authenticated routes. The ThreadWizard dialog inherits the auth context from its parent.
- **No token storage in component state:** The token is read from `AuthContext` per-request, not cached in local state or localStorage by the component.

### 5.4 Rate Limiting (429) Handling

The current API client functions in `lib/api/threads.ts` throw generic errors on non-OK responses. For rate-limited responses, the following enhancement is specified:

**API client layer** (`lib/api/threads.ts`):
- Each function should check for `resp.status === 429` specifically and throw a typed error:
  ```typescript
  if (resp.status === 429) {
    const retryAfter = resp.headers.get('Retry-After');
    throw new RateLimitError(
      'Too many requests. Please wait before trying again.',
      retryAfter ? parseInt(retryAfter, 10) : 60,
    );
  }
  ```
- Define `RateLimitError` class extending `Error` with a `retryAfterSeconds` property.

**UI layer** (ThreadWizard / StepGenerate):
- When the generate mutation fails with a `RateLimitError`, display a specific message: "Rate limit reached. Please wait {N} seconds before generating again."
- Optionally show a countdown timer using `useEffect` + `setInterval`.
- The "Generate Thread" button re-enables after the `retryAfterSeconds` period.

**React Query configuration:**
- The generate mutations already have `retry: false` implicitly (mutations don't retry by default in React Query). This is correct — we do NOT want automatic retry on 429.
- The read queries (families, sizes, spec, recommendation) use React Query's default retry with exponential backoff, which is appropriate for transient errors but not for 429s. The hook layer should configure `retry: (failureCount, error) => !(error instanceof RateLimitError)` to avoid retrying rate-limited requests.

### 5.5 CSRF Protection

- All state-mutating requests (POST `/threads/generate`, POST `/threads/generate/print-optimized`) use `Content-Type: application/json` with a Bearer token. This is not vulnerable to traditional CSRF because:
  1. The `Authorization` header is not sent automatically by browsers.
  2. `Content-Type: application/json` cannot be triggered by HTML forms.
- No additional CSRF tokens are needed.

### 5.6 Data Exposure

- Thread generation responses contain only geometry metadata (face counts, timing). No PII, no secrets.
- Print recommendations contain only engineering data. No PII.
- The 3D preview is generated client-side from spec data — no binary geometry data crosses the network for preview.
- Error messages from the API are displayed to the user. Ensure the backend does not leak internal stack traces or database details in error responses. This is a backend concern (already handled by FastAPI exception handlers) but the frontend should truncate error messages to 200 characters as a defense-in-depth measure.

### 5.7 Content Security Policy Compatibility

- `ThreadPreview3D` uses WebGL via React Three Fiber. This requires:
  - `script-src` must allow the application bundle (already the case).
  - No `unsafe-eval` needed — R3F does not use `eval`.
  - No external resources loaded — the metallic material uses built-in Three.js shaders, not external textures or HDR files.
  - `Environment preset="city"` from drei loads an internal HDR map bundled with drei. If CSP blocks this, the component should catch the error and fall back to basic lighting (ambientLight only). **Recommendation:** Do NOT use `<Environment preset="...">` in `ThreadPreview3D` to avoid this issue. Use only `ambientLight` + `directionalLight`, which is sufficient for a thread preview.

## Appendix B: Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| WebGL not available in user's browser | Low | Medium | ErrorBoundary fallback to spec table; feature is non-blocking |
| R3F Canvas causes test flakiness | Medium | Medium | Full mock of `@react-three/fiber` and `@react-three/drei` in unit tests; skip 3D in E2E |
| Print recommendation API latency > 2s | Low | Low | Skeleton loading state already handled; data fetched asynchronously, does not block navigation |
| Rate limiting on generate endpoint | Medium | Medium | `RateLimitError` class + UI countdown + 429 detection |
| Thread spec missing for rare families | Low | Low | ThreadPreview3D gracefully handles null spec (no render); generate step works without preview |
| CSP blocks drei Environment preset | Medium | Low | Do not use `<Environment>` in ThreadPreview3D; use plain lights only |
| Bundle size increase from Three.js imports | Low | Medium | ThreadPreview3D uses dynamic import (`React.lazy`) so Three.js is code-split; only loaded when user reaches Step 4 |
